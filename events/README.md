# Events

Agroverse cacao **event activations** — the upcoming/operational tier of the
unified event registry.

## For machines (LLMs): start here
- **[`index.json`](index.json)** — the unified, two-tier registry. Read this first:
  - **`upcoming`** — operational events we're actively planning (full metadata: date, venue, host, format, QR codes, status, next milestone, doc paths).
  - **`past`** — published "cacao circle" pages pulled from `agroverse_shop/event-details-registration/` (title, date, location, url, image, description). Many of these dates are placeholder duplicates from page generation, so each carries a `date_confidence` and the tier lists `date_review_needed`.

## How it's structured (upcoming tier)
- Each upcoming event lives in `events/<slug>/` (e.g. `sftechfestjune26/`).
- **`<slug>/event.json`** is the per-event source of truth (machine-readable twin of the checklist header).
- **`index.json`** is generated — **never hand-edit it.** Edit the per-event `event.json` (and/or regenerate the past tier in agroverse_shop), then run:

  ```bash
  python3 events/build_index.py
  ```

  `build_index.py` embeds the past tier from the local `agroverse_shop` sibling checkout if present, else the published raw URL, else leaves it empty.

Each event folder also holds the human docs: `EXECUTION_CHECKLIST.md` (phased, the working surface), `proposal*.md` (plan-of-record), `implementation_roadmap.md`, and sometimes `field_assets.md` (placard/sticker copy).

## Past tier (where it comes from)
The `past` events are generated by **`agroverse_shop/scripts/build_events_json.py`**, which scans `event-details-registration/<slug>/index.html` and writes `event-details-registration/events.json`. Run that script there when the public event pages change, then re-run `build_index.py` here.

## Convention / playbook
The full "how to run a cacao event activation" playbook — `event.json` schema, the phase convention, QR naming (`PREFIX_CC/CT_YYYY → agl4/agl8`), the Apple Reminder/Calendar check-in convention, and the unified-registry build — lives in **`agentic_ai_context/EVENTS.md`**.
